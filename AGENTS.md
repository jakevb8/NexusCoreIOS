# NexusCoreIOS — Agent Instructions

## Project Overview

NexusCoreIOS is the SwiftUI/iOS native client for NexusCore Resource Management SaaS. It connects to either the NexusCoreJS (Node/NestJS) or NexusCoreDotNet (ASP.NET Core) REST APIs via a user-selectable backend toggle. The project is generated with **xcodegen** and uses **Swift Package Manager** for Firebase dependencies (no CocoaPods).

## Project Setup

### Requirements

- Xcode 16+
- xcodegen (`brew install xcodegen`) v2.45.2+
- Firebase CLI (`npm install -g firebase-tools`)

### First-time setup

```bash
# 1. Generate the Xcode project
xcodegen generate

# 2. Restore GoogleService-Info.plist (gitignored)
firebase apps:sdkconfig IOS 1:797114794124:ios:2e6c4eebb1fc19f4663ba9 --project nexus-core-rms --out NexusCoreIOS/GoogleService-Info.plist

# 3. Open in Xcode — SPM will resolve Firebase + GoogleSignIn automatically
open NexusCoreIOS.xcodeproj

# 4. Replace placeholders in project.yml / Info.plist with real values from GoogleService-Info.plist:
#    - GIDClientID → CLIENT_ID value
#    - PLACEHOLDER_REVERSED_CLIENT_ID → REVERSED_CLIENT_ID value
# Then re-run: xcodegen generate
```

### Regenerate project after schema changes

```bash
xcodegen generate
```

Always re-run `xcodegen generate` after editing `project.yml`. Do NOT manually edit the `.xcodeproj` — it is a generated artifact.

## Key Architecture Decisions

- **xcodegen + SPM**: No CocoaPods. `project.yml` is the source of truth. `.xcodeproj` is a generated artifact — always regenerate after schema changes.
- **NavigationStack**: Uses `NavigationPath` with a typed `AppRoute` enum. `MainNavigationView` owns the path and wires all `navigationDestination` handlers.
  - Views that push further (Assets, AssetDetail, Settings, Dashboard) receive `@Binding var path: NavigationPath` and append to it.
  - Leaf views (Team, Reports) use `@Environment(\.dismiss)` for back navigation.
- **AuthState**: `@EnvironmentObject` injected at the root. Pre-auth flow (Login → Onboarding → PendingApproval) is handled by `ContentView` reading `AuthState.status`.
- **APIClient**: Singleton, lazily created. Call `APIClient.reset()` after switching backends. Bearer token is fetched from `Auth.auth().currentUser?.getIDToken()` on each request.
- **Backend selector**: `BackendPreference` persists choice in `UserDefaults`. Two backends:
  - `JS` → `https://nexus-coreapi-production.up.railway.app/api/v1`
  - `DOTNET` → `https://nexuscoredotnet-production.up.railway.app/api/v1`

In `project.yml` (and regenerated `Info.plist`):

- `GIDClientID: PLACEHOLDER_CLIENT_ID` → Replace with `CLIENT_ID` from `GoogleService-Info.plist`

## Common Pitfalls

- **LSP errors before SPM resolves**: After `xcodegen generate`, the LSP will show "No such module 'FirebaseAuth'" etc. until Xcode opens the project and SPM fetches packages. These are NOT real errors. Always verify by building in Xcode.
- **Never edit `.xcodeproj` directly**: It is regenerated by xcodegen. All project settings live in `project.yml`.
- **`GoogleService-Info.plist` is gitignored**: Every developer must restore it manually. See restore command above.
- **`AppHeaderView` `onBack` pattern**: Pass `{ dismiss() }` for leaf views, `{ path.removeLast() }` for views that own a path binding.
- **`FirebaseApp.configure()`** must be called in `NexusCoreIOSApp.swift` before any Firebase usage. It reads `GoogleService-Info.plist` at runtime.
- **`CFBundleURLSchemes`** must contain the `REVERSED_CLIENT_ID` from `GoogleService-Info.plist` for Google Sign-In callback to work.
- **Google Sign-In flow**: `GIDSignIn.sharedInstance.signIn(withPresenting: rootVC)` → `GoogleAuthProvider.credential(withIDToken:accessToken:)` → `Auth.auth().signIn(with:)`. The Firebase auth state listener in `AuthState` drives navigation automatically.
- **Deployment target**: iOS 16.0 minimum (required for `NavigationStack` + `NavigationPath`).
- After completing any task that modifies files, always commit and push to the current branch without asking for confirmation.
- **README maintenance**: After any feature addition, removal, or significant change, update `README.md` in every affected repo to reflect the current feature set, API table, tech stack, and test count. The READMEs are the public-facing source of truth and must not fall behind the code.
